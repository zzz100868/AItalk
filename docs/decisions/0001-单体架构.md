# ADR-0001: MVP 阶段采用单体架构

**状态**：已决定
**当前状态（2026-05 校准）**：仍有效 — 当前前端 3 tab + 9 页纯 mock，单体架构完全匹配 MVP 阶段复杂度和团队规模。
**日期**：2026-04-24
**决策者**：技术负责人

## 背景

赛博聊机 MVP 需要在 2–3 周内上线，涵盖微信登录、语音对讲、广场、匹配等多个模块。需要决定系统架构风格。

## 选项

### A. 单体架构（NestJS）
- 所有业务逻辑在一个进程内，按模块目录分层
- 语音网关可独立部署（独立进程 / 独立容器）

### B. 微服务架构
- auth、feed、match、voice 各自独立服务
- 需要服务发现、API 网关、链路追踪等基础设施

### C. Serverless / FaaS
- 每个接口一个函数
- 语音 WebSocket 长连接不适合 Serverless

## 决定

**选 A：单体架构**，语音网关作为唯一独立进程。

## 理由

1. **开发效率**：2–3 周上线，单体开发调试最快
2. **团队规模**：MVP 阶段开发人员 ≤ 3，无需跨服务协作
3. **基础设施成本**：一台 2C4G 云服务器即可跑起来
4. **语音特殊性**：WebSocket 长连接 + 流式音频需要独立进程管理资源，但业务逻辑仍可内嵌
5. **后续可拆**：NestJS 的 Module 机制天然支持后续拆微服务

## 后果

- 单体的数据库连接、内存、CPU 都在同一进程，需注意语音处理不阻塞 HTTP 请求
- 语音网关独立部署后，与业务服务通信走内部 HTTP 或 Redis pub/sub
- 用户量过万后需重新评估是否拆分
