# ADR-0004: 主数据库选用 PostgreSQL

**状态**：已决定
**当前状态（2026-05 校准）**：仍有效 — 架构文档全部对齐 PostgreSQL + JSONB（profile_documents）+ pgvector（long_term_memories 向量检索），一库方案未变。
**日期**：2026-04-24
**决策者**：技术负责人

## 背景

需要选择 MVP 阶段的主数据库，承载用户、帖子、匹配、画像等全部业务数据。

## 选项

| 方案 | 优势 | 劣势 |
|---|---|---|
| **PostgreSQL** | JSONB 原生支持（画像文档）；pgvector 扩展（记忆向量检索）；强类型 + 强事务；云服务便宜 | 运维复杂度略高于 MySQL |
| MySQL 8 | 最熟悉、生态成熟 | JSON 支持弱于 PG；无原生向量搜索 |
| MySQL + MongoDB | 关系数据 MySQL、文档数据 MongoDB | 两套存储运维成本翻倍 |
| MongoDB alone | 灵活 schema | 不适合关注/粉丝等关系查询 |

## 决定

**选 PostgreSQL，一库承载所有。**

## 理由

1. **JSONB**：`profile_documents`（用户画像）是典型的半结构化文档，PG 的 JSONB 支持索引 + 查询，无需额外引入 MongoDB
2. **pgvector**：长期记忆需要向量检索（语义相似度召回），pgvector 扩展直接在同一实例搞定，不需要单独的 Milvus / Pinecone
3. **事务一致性**：匹配结果写入需要事务（两人同时标记已匹配），关系数据库最可靠
4. **一库运维**：MVP 阶段只维护一个 PG 实例 + Redis，运维成本最低

## 后果

- 需要安装 pgvector 扩展（云 PG 一般默认支持）
- 大规模时（10w+用户）向量检索可能需要迁移到专用向量库
- ORM 用 Prisma 或 TypeORM（NestJS 生态）
