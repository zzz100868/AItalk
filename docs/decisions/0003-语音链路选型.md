# ADR-0003: 语音链路选用火山引擎（豆包）全家桶

**状态**：已决定
**当前状态（2026-05 校准）**：仍有效 — 前端 index 页语音通话为纯 mock（本地计时器，无 WebSocket/ASR/TTS 调用）。架构文档已确认 ASR → 对话编排 → LLM → TTS 全链路设计，火山引擎（豆包）全家桶选型不变，待语音网关开发时接入。
**日期**：2026-04-24
**决策者**：技术负责人

## 背景

产品要求"真实时流式语音对讲"，需要 ASR（语音识别）、TTS（语音合成）、以及 LLM（对话生成）三个模块。需要决定各模块供应商。

## 选项

### ASR

| 方案 | 说明 |
|---|---|
| **豆包流式语音识别 2.0** | 中文识别准确率高、支持流式、有良好的中文口语处理 |
| 阿里云 Paraformer | 开源可自部署，但流式延迟稍高 |
| 腾讯云 ASR | 微信生态友好，但中文口语效果一般 |

### TTS

| 方案 | 说明 |
|---|---|
| **seed-tts-2.0-standard** | 音色自然、中文女声选择多、支持流式合成 |
| 通义语音合成 | 音色偏机械 |
| Azure TTS | 中文音色自然度不如国产 |

### LLM

| 方案 | 说明 |
|---|---|
| **豆包 pro** | 中文对话能力强、同厂商延迟最低、支持 JSON 输出 |
| DeepSeek-V3 | 性价比极高、中文能力出色 | 
| 通义千问-Max | 阿里系，结构化输出好 |

## 决定

**MVP 阶段 ASR + TTS + LLM 统一用火山引擎（豆包）**。

## 理由

1. **同厂商延迟优势**：ASR → LLM → TTS 三段在同一云内网，减少 50–100ms 跨网延迟
2. **统一计费**：一个账号管理，财务简单
3. **中文体验**：豆包在中文口语场景（ASR 识别 + TTS 自然度 + LLM 对话）综合最佳
4. **流式支持**：三段都支持 streaming，可以做到"边识别边生成边合成"

## 后续

- LLM 备选 DeepSeek-V3，在豆包 pro 出问题或成本过高时切换
- LLM #1（抽取模型）可以用豆包 lite 降本
- 需申请火山引擎账号 + 开通 ASR / TTS / LLM 服务 + 拿 API Key
