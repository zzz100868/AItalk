# ADR-0002: 真人认证选用微信实名认证

**状态**：已决策
**当前状态（2026-05 校准）**：采用微信实名认证方案。微信用户已完成实名认证，通过微信登录的用户可直接视为已实名。后端通过 `wx.login` 获取 openid 后，调用微信平台能力确认实名状态，写入 `users.real_name_verified = true`。不再使用腾讯云人脸核身。
**日期**：2026-05-15（修订）
**决策者**：技术负责人

## 背景

产品要求用户必须通过真人认证才能参与匹配，确保平台用户真实性。

## 选项

| 方案 | 优势 | 劣势 |
|---|---|---|
| ~~腾讯云人脸核身~~ | 公安一所数据源；人脸活体检测 | 约 1 元/次，百万用户级成本高；接入复杂；需额外开通服务 |
| **微信实名认证（采纳）** | 零成本；用户体验最好（无额外操作）；微信已强制实名 | 依赖微信生态；无法独立于微信验证 |

## 决定

**采用微信实名认证**，放弃腾讯云人脸核身。

## 理由

1. **零成本**：微信平台已要求用户实名，无需额外付费服务
2. **零摩擦**：用户无需拍照、输入身份证号，登录即认证
3. **足够可信**：微信实名基于银行卡绑定 + 身份证验证，满足社交匹配场景需求
4. **简化架构**：不需要对接第三方 SDK，减少外部依赖

## 后果

- 不存储身份证号等敏感信息，只存 `real_name_verified` 布尔标记
- 后端通过微信登录链路确认实名状态
- accountSecurity 页面展示实名认证状态（已认证 / 未认证）
- 匹配算法中 `real_name_verified` 条件保留，MVP 阶段可暂跳过
