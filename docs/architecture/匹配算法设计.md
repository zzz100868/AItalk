**上游**：[技术方案设计 §4.4](技术方案设计.md) · [66题维度映射](../product/66题维度映射.md)

## 当前前端实现

当前 `pages/match/match.js` 的匹配逻辑：

1. **开放时间**：每周二（`getDay() === 2`），有 `TEST_MODE = true` 开关始终开放
2. **触发方式**：用户手动点击"开始匹配"按钮（`doMatch()`）
3. **前端 mock 逻辑**：从 `mockData.MATCH_CANDIDATES`（6 位候选人）中随机抽 1 位，避免连续匹配同一人
4. **动画流程**：shake(900ms) → glow(1700ms) → reveal(3000ms) → 显示结果
5. **结果展示**：avatar, name, bio, compatibility, tags, icebreakers, matchInsight
6. **付费入口**：`showPayModal` → "解锁微信"弹窗（confirmPay 占位，未实际接入支付）
7. **倒计时**：非开放时段显示到下个周二 00:00 的倒计时
8. **重置**：`resetMatch()` 可清空当前匹配结果

**后端需要做的**：将 mock 随机替换为真实匹配算法，返回格式与当前候选人结构一致。

## 匹配节奏（后端目标）

- **触发时机**：每周二开放（对齐前端 `getDay() === 2`）
- **用户池**：过去 7 天活跃 + 至少完成 1 轮 AI 语音对话 + 高权重维度（D1/D2/D3/D6/D7）置信度均 ≥ 0.4
- **结果数**：每人每周最多匹配 1 位（MVP），后续可扩至 3 位

## 匹配流程

```
[1] 用户池过滤
    - real_name_verified = true（MVP 阶段若认证未上线，暂跳过此条件）
    - 7 天内有活跃（登录 / 通话）
    - 高权重维度置信度 ≥ 0.4
    - 未被封禁
    ↓
[2] 硬条件过滤（per pair）
    - 性别 × 性取向 兼容
    - 同城（city 一致）；后续可扩至"同省"
    - 年龄差 ≤ 用户设定容忍度（默认 ±5 岁）
    - 双方未在过去 4 周被匹配过
    ↓
[3] 打分（per pair）
    → 得到 match_score ∈ [0, 100]
    ↓
[4] 配对（全局）
    → 贪心 / 稳定婚配
    ↓
[5] 生成匹配文案
    → LLM 写理由 + 破冰话题 + insight
    ↓
[6] 写库 + 推送
```

## 打分公式

```
match_score(A, B) =
    w1 × complementary_score(A, B)     # 维度互补
  + w2 × interest_overlap(A, B)        # 兴趣重合
  + w3 × stage_fit(A, B)               # 关系阶段适配
  + w4 × activity_bonus(A, B)          # 活跃度加成
```

### 权重（MVP 初始值，后续数据驱动调优）

| 因子 | 权重 | 说明 |
|---|---|---|
| w1 complementary | 0.40 | 核心 |
| w2 interest | 0.25 | 兴趣 |
| w3 stage_fit | 0.25 | 阶段 |
| w4 activity | 0.10 | 活跃 |

**注意**：旧设计有 w5 behavior_signal（广场行为加分），因广场功能当前未实现，暂不纳入。后续恢复广场后可加回。

### 3.1 维度互补分（complementary_score）

对 D1–D10 各维度：

```
for each dim in dimensions:
    if dim.matching_mode == "similar":
        # 例如 D1(婚恋取向) — 希望双方目标一致
        dim_score = 1 - |A.dim.score - B.dim.score|
    elif dim.matching_mode == "complementary":
        # 例如 D6(沟通风格) — 一方先思考+一方直接 = 互补
        dim_score = |A.dim.score - B.dim.score|
    weight = dim.matching_weight_normalized
    total += weight × dim_score
```

维度匹配模式：

| 维度 | 模式 | 理由 |
|---|---|---|
| D1 婚恋取向 | similar | 结婚意愿要一致 |
| D2 家庭模式 | similar | 丁克 / 养育观分歧大 |
| D3 核心价值观 | similar | 底层价值对齐 |
| D4 生活节奏 | similar | 日常摩擦少 |
| D5 社交风格 | complementary | 互补效果好 |
| D6 沟通风格 | complementary | 互补 |
| D7 关系恐惧 | similar | 恐惧点一致更有共鸣 |
| D8 兴趣图谱 | mixed（重合 60% + 差异 40%）| 有共同话题也有新鲜感 |
| D9 未来规划 | similar | 方向要对齐 |
| D10 社会态度 | similar（低权重） | 次要因素 |

### 3.2 兴趣重合分

基于 D8 下的各题选项，计算 Jaccard 相似度：

```
overlap = |A.interests ∩ B.interests| / |A.interests ∪ B.interests|
```

### 3.3 关系阶段适配分

```
stages = ["随便聊聊", "想认真了解", "想稳定下来", "想结婚"]
stage_fit = 1 - |index(A.stage) - index(B.stage)| / (len(stages) - 1)
```

### 3.4 活跃度加成

```
activity(user) = normalize(
    0.5 × voice_sessions_7d
  + 0.3 × chat_messages_7d
  + 0.2 × logins_7d
)
activity_bonus(A, B) = min(activity(A), activity(B))
```

**注意**：因广场已移除，活跃度计算中 `posts_7d` 和 `comments_7d` 被替换为 `chat_messages_7d`（记忆库聊天活跃度）。

## 配对算法

### MVP：贪心

1. 计算所有合格 pair 的 `match_score`
2. 按分数降序排列
3. 依次取最高分 pair，标记双方已配对
4. 跳过已配对用户
5. 剩余未配对用户本周空缺

### 后续：改进版 Gale-Shapley

- 双方各自对候选列表按分数排偏好
- 跑 Gale-Shapley 得到稳定匹配
- 好处：全局最优且稳定

## 匹配文案生成

配对完成后，异步调 LLM 为每对生成：

**Prompt 模板：**

```
你是赛博聊机的匹配文案师。根据两位用户的画像，写出：
1. 一段匹配理由（2-3 句，要有"故事感"，引用双方的具体偏好或经历）
2. 两条破冰话题（自然、有趣、能引发对话）
3. 一段匹配洞察（1-2句，揭示两人深层共鸣点）

## 用户 A 画像
{{A.profile_summary}}

## 用户 B 画像
{{B.profile_summary}}

## 匹配核心维度
{{top_3_matched_dimensions_with_scores}}

## 输出格式
{"reason": "...", "icebreakers": ["...", "..."], "insight": "..."}
```

**示例输出**（对应前端 mockData 结构）：
```json
{
  "reason": "你们都在深夜最有灵感，也都偏好安静的周末和深度阅读。",
  "icebreakers": [
    "看到你也喜欢在清晨喝手冲咖啡，有什么推荐的豆子吗？",
    "你的主页有一种很安静的力量，周末通常怎么度过？"
  ],
  "insight": "你们同样偏好安静的周末与深度的自我对话，这种对「留白」的共同追求，为建立无压力的灵魂连接提供了土壤。"
}
```

## 冷启动策略

用户 < 50 时：
1. 放宽"同城"为"同省"
2. 降低维度置信度门槛至 0.2
3. 文案增加"你们都是早期用户"之类的破冰引导
4. 保底：每人至少匹配 1 位（即使分数不高，AI 文案包装）

## 匹配反馈闭环

匹配后收集用户反馈 → 反哺画像：

| 反馈类型 | 画像影响 |
|---|---|
| "聊得来" | 强化本次匹配涉及的维度得分 |
| "不太合适" + 选择原因 | 调整对应维度或修正 `ideal_type_hypothesis` |
| "跳过" | 弱负信号，降低类似画像的推荐权重 |
| 持续聊天 >3 天 | 强正信号 |
