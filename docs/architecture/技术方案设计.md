# 赛博聊机 · 技术方案设计 v0.2

**状态**：初稿，待评审
**适用范围**：MVP 阶段的后端与语音链路
**上游文档**：[产品概念文档](../product/产品概念文档.md)

## 0. 设计原则

1. **前端不动**：当前 3 个 tab（匹配 / 记忆库 / 我的）以及 index（语音通话入口）、editProfile、userHome、settings、accountSecurity、notifications 等页面的交互形态全部保留，后端 API 以适配这些页面当前调用的字段和流程为目标。
2. **分层而非端到端**：语音走 ASR → 对话编排 → LLM → TTS，保证 66 题维度覆盖和画像可解释。
3. **MVP 单体优先**：避免上来就微服务，先单体 + 逻辑分层，核心能力模块化，便于后续拆分。
4. **每个画像结论可回溯**：所有 profile 字段都要有 evidence 指向某段对话 / 反馈。

## 2. 架构总览

```
微信小程序 (当前前端)
    │  HTTPS / WSS
    ▼
API 网关 (鉴权 + 限流 + 路由)
    │
    ├── 业务服务 (NestJS) ── MVP 单体
    │     ├─ auth          (微信登录 + 真人认证)
    │     ├─ user/profile  (基础资料 + 恋爱人格卡)
    │     ├─ voice-session (语音对讲会话编排)
    │     ├─ dialogue      (对话编排 / 状态机 / 66题采样)
    │     ├─ memory        (长期记忆 + 画像 + chat)
    │     ├─ match         (每周匹配 + 破冰)
    │     ├─ notification  (通知系统)
    │     └─ payment       (微信支付 + 权益)
    │
    ├── 实时语音网关 (WebSocket)
    │     ├─ 对接豆包 ASR 流式识别
    │     ├─ 对接 seed-tts-2.0 合成
    │     └─ 与 dialogue 模块双向通信
    │
    ├── 异步任务 (队列 + 定时)
    │     ├─ 画像抽取 worker  (对话结束后异步跑)
    │     ├─ 每周匹配 job     (周二定时触发)
    │     └─ insight 生成 worker
    │
    └── 存储
          ├─ PostgreSQL     业务主库
          ├─ Redis          会话 / 限流
          ├─ PG JSONB       画像文档 + 证据
          ├─ 向量库 (pgvector)  记忆检索
          └─ OSS/COS        头像 / 照片 / 语音包
```

**与旧架构的差异**：
- 移除 `feed`（广场）模块和 `social`（关注/粉丝）模块 — 当前前端无入口
- 新增 `notification` 模块 — 对应 notifications 页面
- 匹配 job 时间从周五 20:00 调整为每周二（对应前端 `now.getDay() === 2` 逻辑）

**语言 / 框架**：NestJS 做业务主服务，语音网关用 Node.js 独立进程（见 ADR-0005）。语音并发压力大（>500 同时通话）时可用 Go 重写语音网关。

## 3. 域模型与数据库设计（核心表）

### 3.1 用户域
- `users`：`id, openid, unionid, nickname, avatar, gender, city, birth_year, orientation, bio, real_name_verified, verify_provider, created_at`
- `user_settings`：`user_id, notify_weekly_match, privacy_level, ...`
- `user_photos`：`user_id, photo_url, sort_order, created_at`（对应前端照片墙，最多 8 张）

### 3.2 对话 & 记忆域
- `voice_sessions`：`id, user_id, started_at, ended_at, duration_sec, status(ongoing/ended/timeout), round_no`
- `dialogue_turns`：`session_id, idx, role(user/ai), text, audio_url, asr_confidence, created_at`
- `profile_documents`（每用户一份，JSONB）：完整结构同产品文档 4.3 示例
- `profile_evidence`：`user_id, dimension, evidence_text, source(voice/feedback), session_id, confidence_delta, created_at`
- `covered_questions`：`user_id, question_id, dimension, covered_at, session_id`
- `long_term_memories`：`user_id, kind(preference/stage/event), content, embedding(vector), last_reinforced_at`
- `memory_insights`：`id, user_id, title, content, tag, tag_color, category(life/emotion/hobby/growth), date, created_at`
- `memory_chat_messages`：`id, user_id, sender(user/ai), content, created_at`

### 3.3 题库域
- `questions`：`id, raw_text, dimension, direction(+/-), probe_hints(JSON), extract_rules(JSON), weight`
- `dimensions`：`id, name, label, description, matching_weight`

### 3.4 匹配域
- `match_rounds`：`id, scheduled_at, status(pending/generating/published)`
- `match_results`：`round_id, user_a, user_b, score, reason_text, icebreakers(JSON), insight_text, unlocked_by_a, unlocked_by_b`
- `match_feedback`：`match_id, user_id, sentiment, skip_reason, comments`

### 3.5 通知域
- `notifications`：`id, user_id, type(follow/match/system), author_name, author_avatar, content, read, created_at`

### 3.6 支付域
- `entitlements`：`user_id, kind(deep_chat/unlock_wechat/...), target_match_id, expires_at`
- `orders`：`id, user_id, amount, sku, wx_transaction_id, status`

## 4. 关键模块设计

### 4.1 实时语音对讲（最难的一块）

前端 `pages/index/index` 有 `startCall / endCall / toggleMute / toggleSpeaker`。当前只是本地计时器 + mock UI。**需后端接起来，不改 UI。**

通话结束后前端设置 `getApp().globalData.memoryTargetTab = 'archive'` 并跳转到记忆库页。

**协议**：WebSocket

```
client → server:  {type:"start", session_id?, device_info}
client → server:  {type:"audio_chunk", seq, pcm_base64}  # 40ms 一帧
server → client:  {type:"asr_partial", text}             # ASR 流式回推
server → client:  {type:"ai_reply_audio", seq, pcm_base64, text}
server → client:  {type:"ai_turn_end"}
server → client:  {type:"session_soft_close", reason}    # 到时/信息饱和
client → server:  {type:"extend"}                        # 用户要求"再聊一会儿"
client → server:  {type:"end"}
```

**状态机**（服务端 per-session）：
`idle → listening → asr_streaming → dialogue_thinking → tts_streaming → listening ... → closing → ended`

**调度**：
- ASR：豆包 `volc.seedasr.sauc.duration` 长流式
- TTS：`seed-tts-2.0-standard` 流式合成，边合成边下发（首字 < 500ms）
- LLM：回复在 VAD 检测到用户结束讲话后立即触发
- 打断（barge-in）：用户说话时 VAD 触发 → 停掉正在下发的 TTS

**时长控制**：5 分钟最低、15 分钟最长、软收尾阶段由 dialogue 模块决策。

### 4.2 对话编排 / 66 题隐式采样

详见 [对话编排设计](对话编排设计.md)。

### 4.3 记忆库（memory 页对齐）

memory 页有 3 个子 tab：
- **chat**：与 AI 的文字聊天（当前 mock 回复，后端需接 LLM 对话）
- **memory**：洞察卡片列表（insights），分类 = 全部/生活/情绪/兴趣/成长
- **archive**：人格档案（aboutMe + personalities + traits）

后端需提供：
1. `GET /api/memory/chat` — 历史聊天记录
2. `POST /api/memory/chat` — 发送消息 + 返回 AI 回复
3. `GET /api/memory/insights` — insights 列表（支持 category 过滤）
4. `PUT /api/memory/insights/:id` — 编辑 insight
5. `DELETE /api/memory/insights/:id` — 删除 insight
6. `GET /api/memory/archive` — 人格档案（aboutMe, personalities, traits）

### 4.4 匹配

当前前端逻辑：
- 每周二开放（`now.getDay() === 2`，有 TEST_MODE 开关始终开放）
- 用户点击"开始匹配"→ 从 candidates 池随机抽 1 位
- 展示匹配动画（shake → glow → reveal）→ 结果页
- 结果展示：avatar, name, bio, compatibility, tags, icebreakers, insight
- 支持"解锁微信"付费弹窗（confirmPay 占位）
- 支持复制 icebreaker

后端需提供 `GET /api/match/current` 和 `POST /api/match/do`。详见 [匹配算法设计](匹配算法设计.md)。

### 4.5 真人认证

供应商建议：**腾讯云人脸核身**，理由：
- 微信生态内 `wx.startFacialRecognitionVerify` 原生接口最顺
- 合规（公安一所）
- 小程序 appid 直接绑定，审核快

后端接 `GET /api/auth/face-verify/ticket` → 返回 biz_token，小程序 `wx.startFacialRecognitionVerify` 完成后后端回调验真 → 写 `users.real_name_verified=true`。

## 6. 风险 & 待决策

| # | 事项 | 建议 |
|---|---|---|
| 1 | 端到端语音延迟目标 1–3s | 首字 < 800ms，整句 < 2s，需压测 ASR / LLM / TTS 串接 |
| 2 | 66 题 → 维度映射 | 需产品 + 算法合力把 `66题.xlsx` 拆成 dimensions / questions 表，这是阻塞项 |
| 3 | 人格抽取用大模型成本 | 双模型策略：小模型抽取 + 大模型生成；可考虑批量异步抽取降本 |
| 4 | 微信小程序 WebSocket 限制 | `wx.connectSocket` 单连接、帧大小有限制，语音分片 40ms/帧 + 协议要轻 |
| 5 | 匹配冷启动 | 前两���用户少时，退化为"同城 + 年龄"粗匹配 + AI 文案包装 |
| 6 | 画像隐私 / 合规 | 用户可在设置里控制隐私级别 |
| 7 | 真人认证供应商 | 倾向腾讯云，待 PM 确认预算 |
| 8 | LLM 供应商 | 倾向豆包（和 ASR / TTS 同厂，延迟和计费统一），备选 DeepSeek |
| 9 | memory chat 接入 | 当前 mock 回复，后端需提供 LLM 文字对话能力（与语音共享上下文） |

## 7. 排期建议（MVP 2–3 周）

- **Week 1**：基础设施（鉴权 / DB / 网关）+ 用户 / 资料 / 照片墙 CRUD + 题库导入
- **Week 2**：语音网关 + 对话编排最小闭环（能跑通 ASR → LLM → TTS）+ 画像写入 + memory chat 接入
- **Week 3**：每周匹配 Job + 人格卡生成 + 支付 + 真人认证 + 通知系统 + 联调压测
