# 赛博聊机 · 技术方案设计 v0.1

**状态**：初稿，待评审
**适用范围**：MVP 阶段的后端与语音链路
**上游文档**：[产品概念文档](../product/产品概念文档.md)

## 0. 设计原则

1. **前端不动**：现有 5 个 tab（聊天 / 广场 / 匹配 / 记忆库 / 我的）以及 userHome / editProfile / moments / settings 等页面的交互形态全部保留，后端 API 以适配这些页面当前调用的字段和流程为目标。
2. **分层而非端到端**：语音走 ASR → 对话编排 → LLM → TTS，保证 66 题维度覆盖和画像可解释。
3. **MVP 单体优先**：避免上来就微服务，先单体 + 逻辑分层，核心能力模块化，便于后续拆分。
4. **每个画像结论可回溯**：所有 profile 字段都要有 evidence 指向某段对话 / 帖子。

## 1. 架构总览

```
微信小程序 (现有前端)
    │  HTTPS / WSS
    ▼
API 网关 (鉴权 + 限流 + 路由)
    │
    ├── 业务服务 (Node/NestJS 或 Go) ── MVP 单体
    │     ├─ auth          (微信登录 + 真人认证)
    │     ├─ user/profile  (基础资料 + 恋爱人格卡)
    │     ├─ voice-session (语音对讲会话编排)
    │     ├─ dialogue      (对话编排 / 状态机 / 66题采样)
    │     ├─ memory        (长期记忆 + 画像)
    │     ├─ feed          (广场 / 帖子 / 互动)
    │     ├─ match         (每周匹配 + 破冰)
    │     ├─ social        (关注/粉丝)
    │     └─ payment       (微信支付 + 权益)
    │
    ├── 实时语音网关 (WebSocket)
    │     ├─ 对接豆包 ASR 流式识别
    │     ├─ 对接 seed-tts-2.0 合成
    │     └─ 与 dialogue 模块双向通信
    │
    ├── 异步任务 (队列 + 定时)
    │     ├─ 画像抽取 worker  (对话结束后异步跑)
    │     ├─ 每周匹配 job     (周五 20:00 定时触发)
    │     └─ 帖子理解 worker  (发帖后提取画像信号)
    │
    └── 存储
          ├─ MySQL/PG       业务主库
          ├─ Redis          会话 / 限流 / 排行
          ├─ MongoDB 或 PG JSONB  画像文档 + 证据
          ├─ 向量库 (pgvector)  记忆检索
          └─ OSS/COS        头像 / 帖子图片 / 语音包
```

**语言 / 框架建议**：NestJS（TS，和小程序同语言，团队友好）或 Go（性能更好，语音网关部分并发压力大）。推荐 **NestJS 做业务，Go/Node 单独写语音网关**。

## 2. 域模型与数据库设计（核心表）

### 2.3 题库域
- `questions`：`id, raw_text, dimension, direction(+/-), probe_hints(JSON), extract_rules(JSON), weight`

### 2.4 广场域

> 前端 `explore.js`、`moments.js`、`followers/following` 已有对应交互，字段需对齐（author / avatar / time / content / likes / comments / liked / showComments）。**后端返回时做字段映射**，不改前端。

### 2.5 匹配域
- `match_rounds`：`id, scheduled_at, status(pending/generating/published)`
- `match_results`：`round_id, user_a, user_b, score, reason_text, icebreakers(JSON), unlocked_by_a, unlocked_by_b`
- `match_feedback`：`match_id, user_id, sentiment, skip_reason, comments`

### 2.6 支付域
- `entitlements`：`user_id, kind(deep_chat/unlock_wechat/...), target_match_id, expires_at`
- `orders`：`id, user_id, amount, sku, wx_transaction_id, status`

## 3. 关键模块设计

### 3.1 实时语音对讲（最难的一块）

前端 `pages/index/index` 已有 `startCall / endCall / toggleMute / toggleSpeaker` 接口。**前端现在只是本地计时器，需要后端把这一套接起来，不改 UI。**

**协议**：WebSocket

```
client → server:  {type:"start", session_id?, device_info}
client → server:  {type:"audio_chunk", seq, pcm_base64}  # 40ms 一帧
server → client:  {type:"asr_partial", text}             # ASR 流式回推
server → client:  {type:"ai_reply_audio", seq, pcm_base64, text}
server → client:  {type:"ai_turn_end"}
server → client:  {type:"session_soft_close", reason}    # 到时/信息饱和
client → server:  {type:"extend"}                        # 用户要求"再聊一会儿"
client → server:  {type:"end"}
```

**状态机**（服务端 per-session）：
`idle → listening → asr_streaming → dialogue_thinking → tts_streaming → listening ... → closing → ended`

**调度**：
- ASR：豆包 `volc.seedasr.sauc.duration` 长流式
- TTS：`seed-tts-2.0-standard` 流式合成，边合成边下发（首字 < 500ms）
- LLM：回复在 VAD 检测到用户结束讲话后立即触发
- 打断（barge-in）：用户说话时 VAD 触发 → 停掉正在下发的 TTS

**时长控制**：5 分钟最低、15 分钟最长、软收尾阶段由 dialogue 模块决策。

### 3.2 对话编排 / 66 题隐式采样

核心：**不是一个纯 prompt 喂给 LLM**，而是 **LLM 工具调用 + 状态机**：

```
每一轮用户说话 →
  [LLM #1: 理解 + 抽取]  输出 { intent, emotion, evidence_candidates[{dim, text}] }
  ↓
  [规则引擎] 更新 covered_questions / dimensions / confidence
  ↓
  [规则引擎] 选择下一步策略:
     - 维度覆盖率 < 阈值 → 选一个 next_probe_hint, 告知 LLM 自然引导
     - 维度饱和 → 进入收尾阶段
     - 用户情绪低 → 切换到陪伴而非追问
  ↓
  [LLM #2: 生成回复]  受约束 prompt: (当前阶段 + 软引导方向 + 不能问卷化 + 温柔女声人设)
  ↓
  [TTS]
```

**LLM 选型**：
- 中文能力 + 结构化输出 + 低延迟 → DeepSeek-V3 / 通义千问-Max / 豆包 pro，性价比优先
- 抽取（LLM #1）可以用更便宜的小模型；生成（LLM #2）用对话能力强的
- 必须支持 JSON Mode / Function Calling

**画像抽取写库**：放异步队列，不阻塞下一轮对话。

### 3.3 长期记忆

- **事实型记忆**（如"在北京"、"喜欢手冲咖啡"）→ `long_term_memories` + 向量
- **画像维度**（如 security_need=0.78）→ `profile_documents`
- **事件记忆**（如"上周分手"）→ 时间敏感，带 `last_reinforced_at`，久未提及 decay

**检索**：每次开新会话时召回 top-K 相关记忆塞到系统 prompt，控制 token 预算。

### 3.4 匹配（每周 Job）

周五 20:00 定时任务：
1. 拉取过去 7 天活跃 + 画像置信度达标的用户池
2. 同城 × 年龄带 × 性别 / 取向过滤
3. 打分：`w1 * 维度互补分 + w2 * 兴趣重合分 + w3 * 关系阶段匹配分 + w4 * 活跃度`
4. 稳定婚配 / 贪心配对，避免一人被多次分配
5. LLM 生成匹配理由 + 破冰话题（模板 + 证据填空）
6. 写 `match_results`，20:00 推送（订阅消息）

前端 `match.js` 页已经在展示 `matchAvatar / matchName / matchBio / compatibility / tags / icebreakers`，字段直接对齐即可。

### 3.5 真人认证

供应商建议：**腾讯云人脸核身**，理由：
- 微信生态内 `wx.startFacialRecognitionVerify` 原生接口最顺
- 合规（公安一所）
- 小程序 appid 直接绑定，审核快

后端接 `GET /api/auth/face-verify/ticket` → 返回 biz_token，小程序 `wx.startFacialRecognitionVerify` 完成后后端回调验真 → 写 `users.real_name_verified=true`。

### 3.6 广场

前端已有交互：发帖（文字 + 图片）、点赞、评论、关注 / 粉丝、他人主页。后端只需补齐对应接口 + 存储 + 审核。

**内容审核**：微信 `msg_sec_check` + 图片 `img_sec_check` 必做（小程序合规硬门槛），异步再跑一遍自研 / 云服务文本风险分类。

**画像信号抽取**：发帖后进异步队列，LLM 抽关键词 / 情绪 / 维度证据，写 `profile_evidence`。

## 5. 风险 & 待决策

| # | 事项 | 建议 |
|---|---|---|
| 1 | 端到端语音延迟目标 1–3s | 首字 < 800ms，整句 < 2s，需压测 ASR / LLM / TTS 串接 |
| 2 | 66 题 → 维度映射 | 需产品 + 算法合力把 `66题.xlsx` 拆成 dimensions / questions 表，这是阻塞项 |
| 3 | 人格抽取用大模型成本 | 双模型策略：小模型抽取 + 大模型生成；可考虑批量异步抽取降本 |
| 4 | 微信小程序 WebSocket 限制 | `wx.connectSocket` 单连接、帧大小有限制，语音分片 40ms/帧 + 协议要轻 |
| 5 | 匹配冷启动 | 前两周用户少时，退化为"同城 + 年龄"粗匹配 + AI 文案包装 |
| 6 | 画像隐私 / 合规 | "公开表达（广场）vs 私密记忆（语音）"分库分权限，用户可在设置里关闭广场 → 画像的推断 |
| 7 | 真人认证供应商 | 倾向腾讯云，待 PM 确认预算 |
| 8 | LLM 供应商 | 倾向豆包（和 ASR / TTS 同厂，延迟和计费统一），备选 DeepSeek |

## 6. 排期建议（MVP 2–3 周）

- **Week 1**：基础设施（鉴权 / DB / 网关）+ 用户 / 资料 / 广场 CRUD + 题库导入
- **Week 2**：语音网关 + 对话编排最小闭环（能跑通 ASR → LLM → TTS）+ 画像写入
- **Week 3**：每周匹配 Job + 人格卡生成 + 支付 + 真人认证 + 联调压测
