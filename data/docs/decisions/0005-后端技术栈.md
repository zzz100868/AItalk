# ADR-0005: 后端技术栈选用 NestJS + 独立语音网关

**状态**：已决定
**日期**：2026-04-24
**决策者**：技术负责人

## 背景

需要确定后端开发语言和框架。主要考量：开发效率、语音 WebSocket 处理能力、团队上手速度。

## 选项

| 方案 | 优势 | 劣势 |
|---|---|---|
| **NestJS (TypeScript)** | 和小程序同语言（JS/TS），团队无需语言切换；Module 机制适合单体分层；生态丰富（Prisma/TypeORM/Bull 队列） | 单线程，CPU 密集任务需注意 |
| Go (Gin/Echo) | 高并发性能强、语音网关天然适合 | 团队需要学 Go；开发速度略慢 |
| Python (FastAPI) | AI/ML 生态好 | WebSocket 性能一般；类型系统弱 |

## 决定

**NestJS 做业务主服务；语音网关用 Node.js 独立进程（共享 TS 代码）。**

如果后续语音并发压力大（>500 同时通话），语音网关可以用 Go 重写。

## 理由

1. **一致性**：前端（小程序 JS）+ 后端（NestJS TS）+ 语音网关（Node TS）全栈 TypeScript，类型定义共享
2. **开发效率**：NestJS 的 Module + Service + Controller 分层 2–3 周能出 MVP
3. **异步任务**：Bull + Redis 做画像抽取、匹配 Job 等异步任务，NestJS 原生支持
4. **语音独立**：WebSocket 长连接和 HTTP 请求隔离，避免语音流量影响业务 API

## 技术栈清单

| 层 | 选型 |
|---|---|
| 框架 | NestJS 10+ |
| 语言 | TypeScript 5+ |
| 数据库 | PostgreSQL 16 + pgvector |
| ORM | Prisma |
| 缓存 | Redis 7 |
| 队列 | Bull (Redis-backed) |
| WebSocket | ws / socket.io（语音网关） |
| 容器 | Docker + docker-compose（开发）；云容器服务（生产） |
| CI | GitHub Actions |

## 后果

- Node.js 单线程：语音音频编解码如需 CPU 密集处理，用 Worker Thread 或 native addon
- MVP 后如需水平扩展，NestJS 无状态部署 + Redis session 即可
- 类型定义可以用 shared package 在业务和语音网关间复用
